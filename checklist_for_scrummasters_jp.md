# An Example Checklist for ScrumMasters in Japanese

## **フルタイムのファシリテーター？**

スクラムマスターの適任者は２つか３つのチームを一度に担当できます。もしあなたが、組織のミーティングを開く、タイムボックスを実施する、はっきりとした報告を元にした障害に対処することだけが役割だと思えば、あなたはパートタイムの心遣いでこの役割を得ることができます。あなたの組織がスクラムに持つ期待やベースラインをチームが超えないのであれば、壊滅的なことは起こらないでしょう。

しかし、もし組織が変革し、誰も思ってもみなかったような偉大なことを成し遂げる大いなる時間を過ごすチームをあなたが想像できるならば、あなたは**偉大な**スクラムマスターと考えられます。

偉大なスクラムマスターは、１つのチームを一度に担当します。

私たちは、最初は熱心なスクラムマスターが７人のチームに１人いることを推奨します。

あなたがやるべき仕事を見つけられていないならば、プロダクトオーナー、チーム、チームのエンジニアリングプラクティス、そしてチーム外の組織をふりかえりましょう。誰にも一つの規定もないなかで、私が見たスクラムマスターの典型的なものを概説しました。どうぞそれぞれのチェックボックスに印をつけてください。印の内容は最後のページに記載します。



## **パート１：プロダクトオーナー がやっていることはどう？**

スクラムマスターはプロダクトオーナーの成長を助けながら、プロダクトバックログとリリースプランの維持の良い方法を見つけます。


- [ ] プロダクトオーナーが最後に考えたことは、プロダクトバックログの優先順位ですか？
- [ ] すべてのステークホルダーからの要望と要求はプロダクトバックログに入っていますか？ 思い出して：バックログは**絶対必要**です。

- [ ] プロダクトバックログの中身はマネージメント可能なサイズですか？　アイテムをマネージメント可能な数に維持するために、粒度の小さいものを上に置き、概要的なエピックを下に置きます。プロダクトバックログの一番上から遠すぎるアイテムを過剰に分析することは逆効果です。あなたの要件は、プロダクトを開発している間に、ステークホルダー/顧客との打ち合わせによって変わります。
- [ ] すべての要求（特にプロダクトバックログの一番上から近いもの）は、INVESTとしてよく表現されていますか？（INVEST = 独立している。交渉可能である。価値がある。見積もり可能である。適切なサイズである。テスト可能である。）
- [ ] プロダクトオーナーに対し、**技術的負債**について、またそれをどう避けるかを教育したことがありますか？ 難問の１つは、自動化されたテストを書くことと、リファクタリングすることを、すべてのバックログアイテムの”完了の定義”に書きこむことかもしれません。
- [ ] バックログはすべてのステークホルダーがすぐ見えるように**情報ラジエーター（発信物）**になっていますか？
- [ ] もしあなたがバックログを管理するのに自動的なツールを使っているとすると、誰でもその使い方を簡単に知ることができますか？　自動的な管理ツールは、スクラムマスターの活発な発信がないと、**情報冷蔵庫**になる危険性があります。
- [ ] あなたは誰もにプリントアウトを見せて情報を発信することを助けていますか？
- [ ] あなたは大きなビジョンの図を作って情報を発信することを助けていますか？
- [ ] あなたは、プロダクトオーナーがバックログアイテムを、ふさわしいリリースか優先順に組み込むことを手伝いましたか？
- [ ] 誰もがリリース計画が現実に合っているか知っていますか？　あなたは毎回のスプリントレビューミーティングでアイテムが”完了”と認められたあとに、プロダクト/リリース バーンダウンチャートを誰もに見せても良いかもしれない。プロダクトバックログアイテムの実際に完成したアイテムと新規に追加されたアイテムの変化量を示している図は、スコープ/スケジュールの動向を早期に発見できます。
- [ ] あなたのプロダクトオーナーは、最後のスプリントレビューミーティングのあとに、リリース計画を調整しましたか？ 予定通りの時間内で十分にテストされた製品を出荷する少数のプロダクトオーナーは、毎回のスプリントのリリースを再計画します。これはおそらく、より重要な仕事が発見されたら、将来行う予定のいくつかのリリースは延期することを必要とします


## **パート２：私のチームがやっていることはどう？**

チームメンバーが協働するように仕向けることを励ますあいだに、技術的なタスクについてあなたが迷うリスクがあります。チームについてのあなたの主な責任を考えましょう。

- [ ] あなたのチームは**フロー**状態にありますか？　この状態のいくつかの特徴としては：
  - 明確なゴール。(期待と規範は認識でき、ゴールは達成可能で、人のスキルセットと力量が適切に調整されている）
  - 集中と焦点。注意が制限されたうえでの高いレベルでの集中。
  - 自意識の感覚がなく、行動と意識が融合している。
  - 直接的で、即座のフィードバック。（活動の方向の成功と失敗が明白である。だから行動は必要な調整ができる。）
  - 能力のレベルと挑戦にバランスを保つ。（活動が容易すぎず、難しすぎない）
  - 状況と活動についての個人を管理する感覚。
  - 活動は本質的にやりがいがある。そのため、アクションに努力を要しない。
- [ ] チームメンバーはお互いが好きで、良い休暇を過ごし、お互いの成功を祝福しているように見えますか？
- [ ] チームメンバーはお互い高度な基準に責任感を保ち、成長するようにチャレンジしていますか？
- [ ] チームメンバーはとても居心地が悪いため、議論しない重要な話題/チャンスがありますか？
- [ ] あなたはふりかえりのために進め方と場所を多様に試しましたか？
- [ ] チームはスプリントゴールに集中することができましたか？　たぶん、あなたはこのスプリントでコミットされたプロダクトバックログアイテムの受け入れ基準を再レビューするために、中間スプリントのチェックをするべきです。
- [ ] スプリントタスクボードはチームが実際におこなっていることを反映していますか？　”ダークマター”(議論していないタスクと、１日の仕事量より大きいタスク)に注意してください。　スプリントのコミットメントと関係していないタスクは、コミットメントの障害です。
- [ ] 実現可能で出荷可能な製品インクリメントを開発するために、あなたのチームは十分スキルがまざった3–9人がいますか？
- [ ] あなたのチームのタスクボードは更新されていますか？
- [ ] チームが自己管理している人工物は、チームが使い勝手が良いように、チームに見えていますか？
- [ ] それらの人工物は余計な世話をやくおせっかい屋から適切に保護されていますか？　チームの外にいる人たちからの日々の過剰な調査は、チーム内部の透明性と自己管理を妨げられるかもしれません。
- [ ] チームメンバーはタスクを自発的に行っていますか？
- [ ] 技術的負債を返済の必要性は、完了の定義ではっきりと明示され、徐々にコードは仕事をするのに楽しい場所になっていますか？
- [ ] チームメンバーは役職や肩書に関係なく、集団として仕事に責任があるような態度ですか？(例えばテスト、ユーザー文書など）


## パート３：私たちのエンジニアリングプラクティスの振る舞いはどう？
- [ ] あなたの開発しているシステムは、誰でも（同じチームであれ違うチームであれ）”テスト実行”ボタンが押せ、回帰テスト失敗(これまでは機能していた)を引き起こしたときに容易に調査できるようになっていますか？
- [ ] あなたは、自動化されたエンドトゥエンドのシステムテスト（「機能テスト」という別名）および自動化されたユニットテストの適切なバランスがありますか？
- [ ] チームがシステムテストとユニットテストを書くことは、開発しているシステムと同じ言語ですか？　チームの一部だけが保守できるようなスクリプト言語やキャプチャプレイバックツールを一緒に使うことはチームを向上させません。
- [ ] あなたのチームはシステムテストとユニットテストの間にある有益なグレーゾーンを発見しましたか？
- [ ] だれかが回帰テストを失敗させたとき、CIサーバは自動的にアラーム音を出しますか？このフィードバックループは時間か分に短縮することができますか？( “Daily builds are for wimps.” --ケント・ベック)
- [ ] **すべて**のテストはCIサーバの結果に入っていますか？
- [ ] チームメンバーは前もってすべて詳細設計すること(Big Up Front Design)の変わりとして、継続的な設計とリファクタリングの喜びを発見しましたか？　リファクタリングは『外部の行動を変更せずに内部の構造を変更する』という厳密な定義を持っています。重複したコード、複雑な条件のロジック(余分なインデントや長いメソッド)、下手な名前をつけられた識別子、やりすぎたオブジェクトの依存関係などがある場合はいつでも、リファクタリングは１時間あたり数回起こります。自信を持ってリファクタリングすることは、自動化されたテストがあるときだけ可能になります。リファクタリングをしないと、将来プロダクトを変更するのが難しくなります。とりわけ、悪いコード上で快く仕事をする開発者を見つけることが難しくなります。
- [ ] 各々のプロダクトバックログアイテムのための、あなたの”完了”の定義には、完全に自動化されたテストカバレッジと、リファクタリングは含まれていますか？　テスト駆動開発を学ぶことは、それを達成する可能性を上げます。
- [ ] チームメンバーは、時間のほとんどをペア・プログラミングに使っていますか？　ペア・プログラミングは保守性を大幅に増加させ、バグ発生率を下げることができます。(もし出荷可能な機能よりもコードの行数で測定するのであれば、)ペア・プログラミングは人々の垣根に挑戦的で、しばしば長い時間が取られます。チームメンバーと最初のペアとして働ことを例として、リードしましょう。メンバーの幾人かがこの働き方を好み始めるでしょう。


## **パート４：組織がやっていることはどう？**
- [ ] チーム間でのコミュニケーションは適切な量がありますか？　”スクラム・オブ・スクラム”はこれにただ１つの成功した方法で、まれにベストです。
- [ ] アーキテクチャの境界が離れていても、各々のチームは動く機能を独力で生み出すことができますか？
- [ ] あなたのスクラムマスターたちはお互いに会い、組織的な障害リストを働かせていますか？
- [ ] 組織的な障害を開発ディレクターのオフィスの壁に貼り付けるのは、いつが適切ですか？ コストは、お金、市場に出るまでの時間の損失、品質の損失、機会損失に定量化できますか？(しかしケン・シュウェーバーの誤り「A dead ScrumMaster is a useless ScrumMaster.」から学びましょう)
- [ ] キャリアパスが少ない組織では、チームのゴールと矛盾はないですか？　もしテストの自動化、またはユーザードキュメントを犠牲にしてプログラミングやアーキテクチャの仕事をするためにキャリアのインセンティブがある場合、”いいえ”と答えてください。
- [ ] あなたの組織は、業界紙、他の独立した情報源または業界のリーダーから最高の働き場所として認識されていますか？
- [ ] あなたは**学習する組織**を作っていますか？


## **結び**

もしあなたがこれらのチェックリストのほとんどに印を付けて、まだ日中に時間があるようならば、私はあなたから連絡をいただきたい。
ひとの創造力を作り出すために、あらかじめ準備された型はありません。このチェックリストは、あなたの状況を助けられるかもしれないし、助けられないかもしれません。
あなたがいったん違いを生み出すことができたと理解し始めたら、それをすることを恐れていることを発見するでしょう。それは正しい道にいることのサインです。



# **使用説明**

あなたが現在の(または最も最近の)雇用者が、スクラムっぽい事をしようとしているか、トレーニングをしていて、あなたにこのチェックリストを受けとったならば、どうぞこれをそこであなたが見たものに適用してください。以下のうちの１つをそれぞれのチェックリストにマークしてください。

✓：よくやれている
△：改善できて、私はどのように始めるか知っている
？：改善できるが、どうやって始めたらよいか分からない
N/A：適用できない、または、利点がまったくない

または、もしあなたの現在の(または最も最近の)雇用者がスクラムっぽいことを試みていないとき、以下のうちの１つをそれぞれのチェックリストにマークしてください。

✓：よくやれている、または、うまくやるのは簡単だ
△：挑戦可能で、どう始めたらよいか知っている
？：挑戦可能だが、どう始めたらよいか分からない
N/A：適応できない、または、利点がまったくない

すべてのチェックリストにマークがついたときには、それらがこのチェックリストから出てきたかどうかにかかわらず、２〜６つの組織的障害を、障害リストに断言しましょう。あなたが、最低1%変わる希望を持っている障害を選びましょう。
